# R3F 嵌入方案详细设计（A/B/C + 协同）

本文给出 `tldraw + R3F` 的三种嵌入方案、选型依据，以及从作业原型走向多人协同产品时的演进路径。重点展开方案 B 的坐标映射原理与 undo/redo 融合。

## 1. 背景与目标

目标不是单纯把 3D 渲染出来，而是同时满足：

- 2D 语义正确：选中、平移、缩放、删除都走 tldraw 交互体系
- 3D 姿态可控：`yRotation` 等模型状态可编辑且可回放
- 历史一致性：undo/redo 可稳定回放关键链路
- 可扩展性：从单人原型过渡到多 shape / 多人协同场景

## 2. 三种方案定义与边界

### 2.1 方案 A：每个 shape 独立 Canvas

- 每个 `MeshyModelShape` 内部一个 R3F `<Canvas>`
- shape 生命周期与 WebGL 实例强绑定
- 3D 与 2D 交互隔离天然清晰

优势：

- 实现最快，最适合先跑通 Must-have
- 状态一致性简单，和 shape 历史天然对齐

短板：

- shape 数量上来后，context 与显存压力明显
- 浏览器/GPU 上限下性能波动较大

### 2.2 方案 B：全局单 Canvas + 多视口（推荐的中长期目标）

- 全局仅一个 WebGL Canvas
- 每个可见 shape 对应一个 viewport/scissor 子区域
- 同一帧中按可见 shape 循环渲染

优势：

- WebGL 资源集中管理，可扩展性最好
- 对 5+ / 50+ shape 的性能潜力最大

短板：

- 坐标映射、事件路由、裁剪与历史融合复杂
- 工程实现成本明显高于 A

### 2.3 方案 C：快照 + 按需实时 3D（A/B 之间的过渡形态）

- 默认展示 2D 快照（静态位图）
- 被选中/hover/编辑时切到实时 3D
- 可做“单活跃”或“小活跃池”

优势：

- 在不重构到 B 的前提下显著降低常态渲染成本

短板：

- 快照失效策略复杂（旋转、光照、缩放阈值等）
- 状态同步与视觉连续性容易出问题

## 3. 方案 B 坐标映射原理（核心）

### 3.1 坐标空间

至少涉及三层：

1. `tldraw` 世界坐标（无限画布）
2. 屏幕像素坐标（容器内 DOM 坐标）
3. WebGL 视口坐标（`gl.viewport` / `gl.scissor`）

### 3.2 基本映射公式

已知：

- shape 世界包围盒 `B = (x, y, w, h)`
- 相机 `camera = (cx, cy, zoom)`
- 画布容器偏移 `offset = (ox, oy)`（相对窗口）

先映射到容器像素坐标：

- `sx = (x - cx) * zoom`
- `sy = (y - cy) * zoom`
- `sw = w * zoom`
- `sh = h * zoom`

再加容器偏移得到窗口像素：

- `vx = sx + ox`
- `vy = sy + oy`

然后转换为 WebGL 视口。注意 WebGL 原点在左下，DOM 在左上，需做 Y 翻转：

- `glY = canvasHeight - (vy + sh)`

最终：

- `gl.viewport(vx, glY, sw, sh)`
- `gl.scissor(vx, glY, sw, sh)`

### 3.3 无限画布下的可见性与裁剪

无限画布不意味着无限渲染。每帧应做：

1. 计算当前屏幕可视矩形 `V`
2. 仅保留与 `V` 相交的 shape 集合
3. 对可见集合排序并渲染（可按 z-index / shape id）

建议：

- 初期可线性扫描；后续引入空间索引（如 R-tree）降低查询成本
- 交互中（拖拽/缩放）使用降级渲染策略，交互结束后恢复质量

### 3.4 事件路由与交互归属

原则：2D 交互仍归 tldraw，R3F 仅处理 shape 内部姿态控制。

- 指针命中先交给 tldraw 选中 shape
- shape 内旋转控件才把事件派发到 3D 控制器
- 禁止 R3F 层绕开 store 直接维护“主状态”

## 4. undo/redo 融合策略

统一原则：**tldraw store 是唯一事实来源**。

- 会影响用户可见状态的操作，必须通过 store 命令链
- `yRotation` 写入 `shape.props.yRotation`
- 异步转换完成后的替换操作同样走 store 更新

建议拆分命令边界：

1. 图片上传（一步）
2. 提交转换任务（可不入历史）
3. 任务完成后“图片 -> 模型”替换（一步）
4. 姿态旋转（每次一步）

并在关键异步边界显式设置历史 stop point，避免撤销出现“中间态漂移”。

## 5. 多人协同场景的关键问题与治理

### 5.1 并发编辑同一 shape

风险：

- 远端覆盖本地旋转或尺寸改动

治理：

- shape 级版本号（`version`）+ 乐观并发控制
- 更新时带 `baseVersion`，冲突则拒绝并提示重试/合并

### 5.2 异步任务回写冲突

风险：

- A 发起转换后，B 删除或替换 shape，任务完成仍写回旧目标

治理：

- 回写前校验 `(shapeId, version, status)`
- 目标不存在或版本不匹配则丢弃回写并记录事件

### 5.3 undo/redo 与远端事件交错

风险：

- 本地撤销栈被远端插入事件污染

治理：

- 区分本地可逆命令流与远端同步流
- 本地 undo 只回放本地命令，远端变更按快照/patch 合并

### 5.4 事件乱序与重复投递

治理：

- 事件携带单调序列号与幂等键
- 客户端维护去重窗口并做乱序重排

### 5.5 协同性能

治理：

- 远端光标/Presence 节流
- 批量 patch 合并广播
- 交互中降级渲染（DPR、阴影、后处理）

## 6. 推荐演进路线

### 阶段 1（当前）

- 用 A 稳定交付功能闭环
- 补齐历史一致性与基本测试

### 阶段 2（性能缓解）

- 引入 C：快照 + 小活跃池
- 建立快照失效规则与预热缓存

### 阶段 3（结构升级）

- 迁移到 B：全局单 Canvas + 多视口
- 引入空间索引与统一渲染调度器

### 阶段 4（协同产品化）

- 上线协同事件协议与冲突治理
- 验证多人场景下历史回放一致性

## 7. 最终建议

- 作业阶段继续以 A 为主是正确选择
- 若目标是真实生产能力，建议优先落地 C 缓解性能，再有计划迁移到 B
- 方案 B 的技术关键不在“能不能画出来”，而在“坐标映射 + 状态一致性 + 协同冲突控制”三者同时成立
